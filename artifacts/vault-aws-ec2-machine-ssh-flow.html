Machine-to-machine SSH from EC2 to a target host

This page focuses on the machine-to-machine SSH story: a workload gets a Vault token through its EC2 instance-profile IAM role, Vault issues a short-lived SSH certificate, the client verifies the server with the Vault host CA, and sshd verifies the client cert with the Vault user CA. The AWS part is just the entry step; the main story is the sequence of trust handoffs after that.

1 — End-to-end flow

Sequence of trust handoffs from workload to target host

Read this top to bottom as one session. The AWS step only gets the machine into Vault. After that, the SSH CA flow takes over: Vault signs a short-lived user certificate, the same source machine acting as the SSH client checks the server's host certificate, and the target host checks that machine's user certificate before allowing the session.

The short version

What the machine is proving at each handoff

  • To Vault: “I am the workload attached to this EC2 instance-profile IAM role, so I am allowed to get a Vault token.”
  • To the SSH server: “I hold the private key matching the public key Vault just signed into a short-lived SSH certificate.”
  • To the source machine acting as SSH client: “I am the intended target host, because I presented a host certificate signed by the Vault host CA trusted in known_hosts.”
Scope note

AWS auth is the bootstrap step, not the main event

Vault documents the iam auth type as the recommended AWS approach for instance-profile workloads.

For this explainer, the important output of that step is simple: the machine gets a Vault token that authorizes one narrow SSH signing action.

2 — How the machine gets a Vault token

A short AWS identity step before the SSH flow begins

An EC2 instance profile gives the workload short-lived AWS credentials through IMDSv2. The workload uses them to authenticate to Vault's AWS auth method, and Vault turns that AWS identity into a Vault token. After that, the rest of the story is ordinary SSH certificate issuance and verification.

Keep this mental model simple
01

The workload receives temporary AWS credentials automatically

Because the role is attached as an EC2 instance profile, there is no need to bake a long-lived AWS secret into the machine.

02

The machine authenticates to Vault as its IAM role

Vault validates AWS-signed identity proof and maps that principal to a Vault auth role.

03

Vault returns a token scoped to one SSH signing path

That token should usually allow one narrow action, such as requesting a short-lived user certificate for a specific machine principal.

Enough detail to be accurate
  • Vault recommends the iam auth type for AWS workloads.
  • Use X-Vault-AWS-IAM-Server-ID to prevent replay between environments.
  • Bind Vault auth roles to the narrowest practical IAM principal ARNs.
  • Keep the follow-on SSH signing role separate from the AWS auth role.
instance profile Vault token bootstrap least privilege
One concrete example is enough
Vault auth role: prod-m2m-ec2
Bound AWS principal: arn:aws:iam::123456789012:role/prod-app-role

That is the only AWS detail you really need to hold in your head: the EC2 workload proves its IAM identity to Vault, and Vault turns that into a short-lived token with permission to request one narrow SSH certificate.
3 — SSH trust chain after Vault auth

After Vault auth, the SSH part is CA-based on both sides

Once the workload has a Vault token, it asks Vault to sign its SSH public key through the SSH secrets engine. From then on, OpenSSH handles the connection exactly like any other SSH certificate flow: client verifies the host cert, server verifies the user cert, and both sides keep their private keys local.

Client-side trust

Host verification via Vault host CA

The client trusts a CA in known_hosts, not one raw host key per machine. That means any host presenting a valid Vault-signed host certificate for the matching DNS name is trusted.

@cert-authority *.prod.internal ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQ...
Server-side trust

User verification via Vault user CA

The server trusts the Vault user CA once through TrustedUserCAKeys. It no longer needs every machine public key copied into authorized_keys.

TrustedUserCAKeys /etc/ssh/trusted-user-ca-keys.pem
HostKey /etc/ssh/ssh_host_ed25519_key
HostCertificate /etc/ssh/ssh_host_ed25519_key-cert.pub
Signing role policy

What Vault should constrain

  • principal: only the Unix account the workload really needs
  • TTL: usually minutes for automation
  • extensions: omit permit-pty for non-interactive jobs
  • environment: separate signing roles per trust domain
4 — Sample issued certificate

What the workload actually presents to sshd

A short-lived SSH certificate makes the authorization visible on the credential itself. For machine-to-machine access, the sample below intentionally omits permit-pty; interactive shell access is not a good default for automation.

Representative sample
/etc/ssh/m2m/id_ed25519-cert.pub:
        Type: ssh-ed25519-cert-v01@openssh.com user certificate
        Public key: ED25519-CERT SHA256:2Xw6f5Y0wQ5kB8d8m2l0nS6C1Q1p0mQ2+AbCdEfGhI0
        Signing CA: RSA SHA256:8d1m7VxqQe2x6Qk3bH5mM2Y2c1K0pR9sN4uT1eL7wA0
        Key ID: "prod-app-role:i-0abc1234def567890:deploy"
        Serial: 1725029102456912345
        Valid: from 2026-03-26T00:05:00 to 2026-03-26T00:20:00
        Principals:
                deploy
        Critical Options: (none)
        Extensions: (none)
How to read it
  • Signing CA: the server must trust this CA via TrustedUserCAKeys.
  • Key ID: useful for audit context; many teams encode workload role or instance metadata here.
  • Valid: the compromise window is minutes, not days.
  • Principals: which Unix login names the workload may use.
  • Extensions: empty here on purpose, which means no PTY by default for this automation example.
If the workload needs tunnels or an interactive shell, add only the exact extension required. The cert should describe the least-powerful SSH behavior that still makes the job work.
5 — Hardening notes

What the most secure production version usually looks like

The strongest pattern is not “rotate every SSH key constantly.” It is “keep private keys local and protected, but rotate the authorization layer aggressively.” That gives you strong identity continuity with fast revocation-by-expiry.

Identity side

Tight AWS-to-Vault mapping

  • One instance-profile IAM role per workload trust boundary when possible.
  • Vault auth role bound to the narrowest principal ARN set.
  • X-Vault-AWS-IAM-Server-ID enabled on every environment.
  • CloudTrail monitoring for IAM trust policy changes and unusual AssumeRole patterns.
SSH side

Least privilege in the cert

  • Short user-cert TTLs such as 5–15 minutes for automation.
  • Separate SSH signer roles per environment and per machine persona.
  • No PTY, no forwarding, no extra extensions unless the job needs them.
  • Host keys generated on the target host, with host certs signed by a separate host CA.
When EC2 auth may still matter

Use the older EC2 auth type only for EC2-specific controls

If you need to bind access to AMI IDs, role tags, or other EC2-instance-specific properties, the ec2 auth type can be worth the extra complexity. Otherwise, IAM auth is the simpler and recommended route for instance-profile machine identity.
